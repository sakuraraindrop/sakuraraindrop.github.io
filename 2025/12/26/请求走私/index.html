什么是HTTP请求走私？

HTTP 请求走私是一种干预网站处理接收到 HTTP 请求序列的方式的技术 一个或多个用户。请求走私漏洞通常至关重要，使攻击者能够绕过安全控制，获得利益 未经授权访问敏感数据，并直接威胁其他应用用户。

请求走私主要与HTTP/1请求相关。然而，支持 HTTP/2 的网站可能存在漏洞，这取决于其后端架构。

HTTP请求走私最早于2005年被记录，并因PortSwigger对该主题的广泛研究而重新流行起来。详情请参阅以下白皮书：

  • HTTP Desync Attacks: Request Smuggling Reborn
  • HTTP/2: The Sequel is Always Worse
  • Browser-Powered Desync Attacks: A New Frontier in HTTP Request Smuggling

HTTP请求走私攻击会发生什么？

今天的Web应用程序经常在用户和最终应用程序逻辑之间使用一系列HTTP服务器。用户将请求发送到前端服务器（有时称为负载均衡器或反向代理），然后该服务器将请求转发到一个或多个后端服务器。这种类型的架构在现代基于云的应用程序中越来越普遍，在某些情况下甚至是不可避免的。

当前端服务器将 HTTP 请求转发到后端服务器时，它通常会通过同一个后端网络连接发送多个请求，因为这样效率更高，性能更好。该协议非常简单；HTTP 请求是一个接一个发送的，接收服务器必须确定一个请求结束的位置以及下一个请求开始的位置：

HTTP请求走私漏洞是如何产生的？

大多数 HTTP 请求走私漏洞都是因为 HTTP/1 规范提供了两种不同的方法来指定请求的结束位置：Content-Length 头和 Transfer-Encoding 头。

许多安全测试人员不知道在 HTTP 请求中可以使用分块编码，原因有二：

  • Burp Suite 会自动解包分块编码，以便更容易查看和编辑消息。
  • 浏览器通常不会在请求中使用分块编码，它通常只出现在服务器响应中。

由于 HTTP/1 规范提供了两种不同的方法来指定 HTTP 消息的长度，因此单个消息可能同时使用这两种方法，从而导致相互冲突。规范试图通过说明，如果同时存在 Content-Length 和 Transfer-Encoding 头，则应忽略 Content-Length 头来防止此问题。这可能足以在仅使用单个服务器的情况下避免歧义，但在两个或更多服务器串联使用时则不然。在这种情况下，可能出现问题的原因有两个：

  • 一些服务器不支持请求中的 Transfer-Encoding 头。
  • 一些支持 Transfer-Encoding 头的服务器，如果该头被以某种方式混淆，就可能不会处理它。

如果前端服务器和后端服务器在处理（可能混淆的）Transfer-Encoding 头时表现不同，那么它们可能会对连续请求之间的边界产生分歧，从而导致请求走私的漏洞。

端到端使用 HTTP/2 的网站本质上对请求走私攻击具有免疫性。由于 HTTP/2 规范引入了一个用于指定请求长度的单一、健壮的机制，攻击者无法引入所需的模糊性。
然而，许多网站拥有一个支持 HTTP/2 的前端服务器，但将其部署在仅支持 HTTP/1 的后端基础设施之前。这意味着前端实际上必须将收到的请求转换为 HTTP/1。这个过程被称为 HTTP 降级。

如何执行HTTP请求走私攻击？

经典的请求走私攻击涉及在单个 HTTP/1 请求中同时放置 Content-Length 头和 Transfer-Encoding 头，并通过操控这些头使前端和后端服务器对请求的处理方式不同。这种操作的具体方式取决于两个服务器的行为：

  • CL.TE：前端服务器使用 Content-Length 头，后端服务器使用 Transfer-Encoding 头。
  • TE.CL：前端服务器使用 Transfer-Encoding 头，后端服务器使用 Content-Length 头。
  • TE.TE：前端和后端服务器都支持 Transfer-Encoding 头，但可以通过以某种方式混淆该头，使其中一台服务器不处理它。

这些技术仅能通过 HTTP/1 请求实现。浏览器和其他客户端，包括 Burp，在与在 TLS 握手期间明确声明支持 HTTP/2 的服务器通信时，默认使用 HTTP/2。
因此，在测试支持 HTTP/2 的站点时，您需要在 Burp Repeater 中手动切换协议。您可以在 Inspector 面板的请求属性部分进行此操作。
