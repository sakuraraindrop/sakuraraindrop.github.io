关键函数分析

FUN_40031b34

由后面的分析反推出来param_1指向主导头 Primary Header 的起始地址，这个函数里面对于版本号(2bit)，通过标志(1bit)，内务命令标志(1bit)，空闲位(2bit)，航天器识别字(10bit)，帧长(10bit)进行了解析校验。param_2应该是帧长。

步骤 1：检查版本号（bits 7..6）

  • *param_1 是第一个 header 字节。
  • *param_1 >> 6 取出高 2 位：[7:6]。
  • 要求这两位 == 2（也就是二进制 10），说明 Primary Header 的版本号固定要求为 2。

如果不等于 2：

  • 写错误码 DAT_400aa276 = 0xFC
  • 返回失败（local_b = -1），高 32 位设为 &DAT_400aa270。

步骤 2：检查通过标志（bit 5）

  • (*param_1 >> 5) & 1 取 bit5。
  • 要求这个 bit == 0，表示 Bypass = 0。

否则：

  • DAT_400aa276 = 0xFB
  • 失败。

步骤 3：检查内务命令标志（bit 4）

  • (*param_1 >> 4) & 1 取 bit4。
  • 要求这个 bit == 0，表示这是一个 TC（遥控）帧，而不是 TM（遥测）。

否则：

  • DAT_400aa276 = 0xFA
  • 失败。

步骤 4：检查空闲位（bits 3..2）

  • (*param_1 >> 2) & 3 取 bit3..2（两位）。
  • 要求这两位必须是 0。

否则

  • DAT_400aa276 = 0xF9
  • 失败。

步骤 5：检查航天器识别字

  • *param_1 & 3 取第一个字节的最低 2 bit（bit1..0）。
  • 左移 8 位：(*param_1 & 3) << 8，形成高 2 bit（bit9..8）。
  • param_1[1] 是第二个字节，作为低 8 bit（bit7..0）。
  • 两者按位或，组成 10-bit 的航天器识别字
  • 要求航天器识别字 == 0x013D。

否则

  • DAT_400aa276 = 0xF8
  • 失败。

步骤 6：检查帧长是否与实际长度一致

  • param_1[2] & 3 取第三个字节（byte2）的低 2 bit（bit1..0），它是帧长的高 2 bit。
  • param_1[3] 是第四个字节（byte3），是帧长的低 8 bit。
  • 合并并 +1： 这是 Primary Header 里定义的“帧长度 + 1”的形式（常见 TC 协议中有“(len + 1)”这样的写法，用来表示实际字数）。

然后拿这个 frame_len 和 param_2 比较：

  • 如果 frame_len == param_2：
    • 视为长度正确：
      • local_b = 1 （成功）
      • puVar1 保持为 frame_len
  • 如果不相等：
    • 长度错误：
      • DAT_400aa276 = 0xF6
      • local_b = -1
      • puVar1 = &DAT_400aa270

步骤 7：返回

  • 高 32 位：puVar1
    • 成功：帧长度 + 1
    • 失败：&DAT_400aa270
  • 低 32 位：local_b
    • 1：成功
    • -1：失败

也就是说对于FUN_40031B34分析后，我们已经可以确定主导头中版本号，通过标志，内务命令标志，空闲位，航天器识别字，帧长的校验逻辑。目前还没看到虚拟信道识别字和帧序列号的处理逻辑。

步骤 1：初始化

DAT_400a41a0 是一个接收超时/轮询计数器：最多尝试 0x8000 次读取字节。

步骤 2：从硬件寄存器读 byte，喂给帧解析状态机

这里 _DAT_22018000 很可能是 串口 / 通信接口接收寄存器。
DAT_400a41a2 保存最近一个字节（可能用于调试/日志）。

关键是这一句：

  • PTR_DAT_400aa290：指向了一个接收缓冲区；
  • &DAT_400a41a4：输出帧长度；
  • DAT_400a41a2：当前新收到的一个字节；

因此 FUN_40031fe0 是一个按字节喂入的帧解析状态机：

  • 每次喂一个字节；
  • 全局状态记录已接收的字节数、当前帧长度等；
  • 当一整帧收完时返回 1；
    （错误返回负数；未完成返回 0）。

所以这一段循环的整体语义是：

  • 在 0x8000 次轮询机会内，从串口一点点读数据给状态机，
  • 直到收到一个完整的“注入数据遥控帧”，或者超时退出。

下面对FUN_40031fe0详细分析一下

FUN_40031fe0

几个全局变量：

  • DAT_400ab808：当前已接收的字节计数 / 状态；
  • DAT_400ab814：当前帧的 长度（字节）；
  • DAT_400ab80e：当前写入到 buffer 的索引；
  • 错误码写到 DAT_400aa276：
    • 0xFF：起始字节不符合预期；
    • 0xFE：长度标记字节不合法；
    • 0x81：字节数超出最大允许（0x204）还没收完。

输入输出

输入参数

  • param_1: 缓冲区指针，收集“长度字段之后”的所有帧字节
  • param_2: 输出：帧长度（来自长度标记字节解析）
  • param_3: 当前收到的一个字节

输出

  • 高 32 位：param_2
  • 低 32 位：local_a
    • 0: 正常继续，帧还没收完
    • 1: 收到一整帧
    • <0: 出错（错误码见 DAT_400aa276）

帧格式

  • 第 1 字节：必须是 0x36（字符 ‘6’），否则错误 0xFF；
  • 第 2 字节：
    • 正常情况下必须是 0x81（-0x7F），否则：
      • 如果也是 0x36，则认为又是一个新的起始字节，状态回到 1；
      • 否则错误 0xFF。
  • 第 3 字节：长度标记，交给 FUN_400319f4 去解析，只有特定几种值合法。
    • 如果 FUN_400319f4 认为非法，且这个字节是 0x36，则同样视为新帧起点；
      否则错误 0xFE。
    • 如果合法，DAT_400ab814 被设置为具体长度（16 / 64 / 128 / 256 / 512），并通过 *param_2 回传；
  • 从第 4 字节起，才正式把数据写入 param_1 指向的 buffer。

根据附件3里的注入数据遥控帧格式，我们能知道第 1 字节和第 2 字节就是地址同步字，第 3 字节是方式字

这里 FUN_400319f4 可以直接当成一个查表：

  • 0x5A → 长度 = 0x10 (16 字节)
  • 0x69 → 长度 = 0x40 (64 字节)
  • 0x99 → 长度 = 0x80 (128 字节)
  • 0xA5 → 长度 = 0x100 (256 字节)
  • 0xA9 → 长度 = 0x200 (512 字节)
  • 其他值 → 长度 = 0，返回 -1 表示非法。

最后总共接收的字节数到达 3 + DAT_400ab814 时，说明完整帧收完。这里也告诉我们这里收到的完整帧就是赛题里面标黄要重点解析的部分，后面CRC没有收进去

关键全局变量与硬件寄存器

  • DAT_400aa840：uplink_prefer_flag

    • ‘U’(0x55) 表示“优先尝试通道 A（_22018000），失败再试 B”
    • 0xAA(-0x56) 表示“优先尝试通道 B（_22019000），失败再试 A”

  • DAT_400aa268：enable_chanC_flag

    • 为 0xAA 时允许使用通道 C（_22007040，明文）作为“备用/内部注入”通道；否则不尝试。

  • DAT_400aa270：tc_ok_count

    • 成功接收并进入 40032a60 处理的一次“明文间接指令帧”计数器（累加）。

  • DAT_400aa274：last_tc_seq

    • 最近一次处理帧的“帧序列计数”（Primary Header 第 5 字节），取自 buf[7]，详见第 4 节偏移解释。

  • DAT_400aa275：last_tc_service

    • 最近一次处理帧的数据域第 1 字节，通常作为“服务号/命令族（或命令码）”，取自 buf[8]。

  • DAT_400aa7b8 / DAT_400aa80c：linkA_errCounter / linkB_errCounter（建议）

    • 在 A/B 成功场景中被清零，失败/回退时可能累加（在本函数中只看到清零行为）。

  • DAT_400aa79c / DAT_400aa7f0：enable_linkB_backoff / enable_linkA_backoff（建议）

    • 当对应值为 0xAA（-0x56）时，在 A 或 B 成功后分别预置一个 backoff 计数（-0x8000/-0x7000）到 DAT_400a420a（通过 FUN_40068ca0），用于链路择优/消抖。

  • DAT_400aa394：enable_sched（0xAA 表示打开）

    • 控制下方“基于时间戳的调度循环”是否运行。

  • DAT_400aa308 / DAT_400aa30c：sched_base / sched_window（建议）

    • sched_base：以 FUN_4004eec8 刷新的参考时间/计数
    • sched_window：选择比较阈值的跨度（<0x1389、<0xF2EB8、其他三档，映射到 base-1/base/base+1）

对照注入数据遥控帧格式

  • 外层（注入数据遥控帧）：[引导码(48B)] + [地址同步字(2B)] + [方式字(1B)] + [遥控注入数据(可变)] + [外层CRC(2B)]
  • 遥控注入数据 = [Primary Header(5B)] + [数据域(9/57/121/249/505)] + [差错控制域(2B，异或/求和)]

在本程序的状态机（FUN_40031fe0 / 40032360 / 400326e0）里，外层“地址同步字”与“方式字”先被验证，然后根据“方式字”映射出“注入数据长度”，再对“遥控注入数据”进行校验并将“明文间接指令帧起始指针”保存到全局指针（即 PTR_DAT_400aa29c / PTR_DAT_400aa298）。

因此：

  • PTR_DAT_400aa29c 指向“整个外层帧的起点”或“至少包含外层 2B sync + 1B mode 的缓冲起点”。这样：
    • 偏移 0..1：地址同步字（常量 0x36, 0x81）
    • 偏移 2：方式字（1 字节，见下节映射）
    • 偏移 3..7：Primary Header（5 字节）
    • 偏移 7：Primary Header 第 5 字节＝帧序列计数（seq）
    • 偏移 8：数据域第 1 字节（通常是服务号/命令族/子服务）
  • 本函数在成功接收一帧后，取：
    • DAT_400aa274 = PTR_DAT_400aa29c[7] → “帧序列计数”
    • DAT_400aa275 = PTR_DAT_400aa29c[8] → “数据域第 1 字节（服务或命令族）”
  • 对第三通道同理：DAT_400aa274 = PTR_DAT_400aa298[7]；DAT_400aa275 = PTR_DAT_400aa298[8]。

定时调度段（DAT_400aa394 == 0xAA 的 while 循环）的宏观语义

  • 这是一个“小型时间片/节拍调度器”，在本主循环中“条件开启”。
  • 它每次调用 sched_tick() 刷新系统计数，再用 sched_prepare() 拉取“目标时间锚点”的 4 个字节（DAT_400a41c9..cc 拼成的 32 位数）。
  • 根据 DAT_400aa30c（门限/周期设定），选择比较值 DAT_400a420c = base-1/base/base+1（base 来自 DAT_400aa308）。
  • 若“当前”达到/等于目标锚点（DAT_400a420c == 目标stamp），则触发 sched_event_fire()（传入 &DAT_400a41cd，疑似推进锚点或执行回调），并累加一次“事件触发计数高 16 位”（DAT_400aa398.2_2++）。每轮末尾用 sched_commit() 提交。
  • 若尚未到点，就自增一个“尝试计数”DAT_400aa3a0（byte），再 sched_commit() 以结束当前尝试。
  • 宏观上，这段像是“按配置窗口对齐时刻、触发排程的心跳/回调”，和遥控主环集成到一起。

结尾日志与计数（TTCmd:%d %d %d %d）

  • DAT_400aa23c：last_src_mark（来自 DAT_400a4180 的某状态/来源码）→ 第一个数字

  • DAT_400aa268 == 0xAA 表示允许通道 C（内部/明文）接入。

其它观察结果

  1. 宏观作用
  • 这条链是“系统上电 → 运行时/驱动初始化 → 启动 appMain → 注册各服务族 → 创建各应用任务 → 启动遥控任务”的任务/调度启动路径。各层职责：
    • 400045bc：早期入口（可选清区）
    • 40004454：底层初始化并调度 40004500
    • 40004500：调度框架/OS 收尾，创建 appMain(4000dd04)
    • 4000dd04：系统与模块大初始化（含 40031f40 注册分发表），然后创建应用任务
    • 4000e0f4：创建 TCmd 等任务，TCmd 的入口是 4001187c
    • 4001187c：遥控任务主循环（真正处理注入帧的地方）
  1. 此链路是否直接处理/修改“注入数据遥控帧”内容？
  • 结论：没有。
    • 在 400045bc / 40004454 / 40004500 / 4000dd04 / 4000e0f4 中，我们没有看到任何对以下对象的访问：PTR_DAT_400aa290/294/298/29c、方式字映射（400319f4）、外层 CRC 校验缓冲、Primary Header 位字段解析、VCID/SCID/帧序号等。也未见对注入帧缓冲的 memcpy/位运算/mask/比较常量。
    • 唯一与注入体系相关的是 40031f40：它设置“服务族/分派表”的初始化开关 DAT_400aa276=1，并调用各服务族初始化函数（与我们之前映射出的时间/串口/VCID 分发族一致），但这属于“安装路由/回调表”的准备工作，不涉及帧具体字段处理。
  1. 任务分配与 FUN_4001187c 的关系
  • 4000e0f4 通过 RTOS 接口 FUN_40073198 创建任务“AppTaskTCmd”，其任务入口函数指针就是 FUN_4001187c。也就是说，FUN_4001187c 作为独立任务运行。
  • 在 400045bc→…→4000e0f4 期间，未见设置 uplink_prefer_flag（DAT_400aa840）或 enable_chanC_flag（DAT_400aa268）等与遥控任务内部分支直接相关的全局；这些标志应在更早的默认值或其他配置路径设置（本链未调整这些值）。
  1. 是否存在“额外的安全检查 / CRC / 认证逻辑”？
  • 这条链路上未出现新的 CRC、认证或白名单检查。帧相关校验仍由已知模块承担：
    • 外层注入帧三状态机：FUN_40031fe0/40032360/400326e0（校验同步字、方式字→长度、差错控制域）
    • Primary Header/差错控制与 VCID 分发：FUN_40031d24/40031b34/40032a60
  • 因此：除上述既有环节外，400045bc→…→4001187c 链路没有新增的“帧内容处理/检查”。

小结：

  • 授时命令的“有效参数”只有 4 字节（mode=0）或 5 字节（mode=1），分别是：类型字(0x0B)、两个相等的校验字节、mode。本层代码并未读取/传递任何后续“具体时间数值”到下游封装。
  • 这强烈说明：星务设置时间可能是“两步法”（先用其他命令把秒值写入某缓存，再用 0x33/0x01/0x0B/mode 触发应用），或时间来源于同步输入（例如地面已经用其他 VCID写了时间寄存器）。但就“如何填写授时指令的数据域”而言，这一层面已经足够给出在本系统里“授时触发帧”的精确定义。

FUN_4005e908

该函数在上一层被这样调用

也就是说它的第一个参数其实是数据域的长度，也就是数据域第2字节的值，第二个参数是指向数据域第3字节的指针，第三个参数是一个指向一片缓冲区的指针。

这层把传进来的 P 的“前 length 字节”（length = mode+4）拷入一个临时区进行对齐/填充，然后附上固定头，写往 PTR_DAT_400ab000，供发送函数使用。关键点：

  • FUN_40016b9c(length, P, &DAT_400abf20)：把 P 的前 length 字节拷贝到 &DAT_400abf20，不足 7 字节时用 0xAA 填充至 7；随后 FUN_4004d17c(…, length) 再写一个字节（用于内部帧结构完善）。
  • 随后把固定 5 字节头（常量区 LAB_4007a5e0）和上面得到的 8 字节缓冲一起拷入 PTR_DAT_400ab000。
  • 实际含义：授时命令被星务“重新打包”为一个带固定头的 8 字节结构，里面包含 0x0B、那两个相等字节、mode。之后由 0xE4 类别通道去确认“执行/触发是否成功”。

回顾三层检查：

  • 顶层 FUN_400650c4：
    • 子命令 subcmd 要为 0x01（这是 VCID handler 的第一个参数 local_b）；
    • DataField[0]（我们称 total_len）必须等于 DataField[4]（mode）+ 4；
    • total_len < 0xF8。
  • FUN_400651d0 对 DataField[1..] 的要求（param_2+1 = P）：
    • P[0] == 0x0B；
    • P[1] == P[2]（两个相等校验字节）；
    • P[3] < 2（mode=0 或 1）；
    • 发送时只取 P 的前 (mode+4) 字节参与封装与发送。

因此，完整数据域布局（以 DataField[0] 起算）为：

  • DataField[0] = total_len = mode + 4
  • DataField[1] = 0x0B // 类型字
  • DataField[2] = X // 任意 1 字节，但 DataField[2] == DataField[3] 必须成立
  • DataField[3] = X // 与 DataField[2] 相等（两个相等校验字节）
  • DataField[4] = mode // 0 或 1（两种触发模式）
  • DataField[5..] = 可选负载（当前链路未用到）
